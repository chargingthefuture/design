## 1. Mission (Non-negotiable)

Skills Hunt is a **gamified community contribution system** that grows the Directory plugin from ~60 to 384+ skilled survivor profiles for a peer-to-peer marketplace serving 5 million human trafficking survivors. Players submit public Quora profile URLs of talented individuals, earn points for valid matches, compete on leaderboards, and win prizes. **No automated scraping** — only user-submitted URLs and manual admin verification.

The plugin is a growth engine for the **Directory plugin** (the canonical list of survivors). Skills Hunt creates "unclaimed" Directory profiles via community submissions; when the real person eventually joins the app, they claim the profile.

## 2. User-locked decisions (do NOT re-ask)

These have been explicitly answered by the owner in earlier sessions. They are binding.

### 2.1 Skills submission field
The submission form provides a **multi-select dropdown populated from the existing skills taxonomy** (`skills_taxonomy_skills` table, joined through `skills_taxonomy_job_titles` → `skills_taxonomy_sectors`). The user picks from the dropdown. **If they don't see what they need**, an optional secondary text input lets them paste/type free text that is parsed into chips by splitting on commas and newlines.

Storage:
- Matched-to-taxonomy selections → `skills_hunt_submissions.skills JSONB[]` (existing column, stores skill names).
- Unmatched free-text tokens → `skills_hunt_submissions.proposed_skills JSONB[]` (new column). Admin can later promote these into the taxonomy.

Rationale: minimizes manual matching by the owner on accept. The taxonomy is the spine.

The original spec's "1–140 chars" single text blob is **superseded**. The owner explicitly chose the taxonomy-first design.

### 2.2 Username system (`@handle`)
- Usernames are handled by **Clerk**. At app signup, every user must create a username through Clerk. This is enforced at onboarding.
- For **unclaimed Directory profiles** generated by Skills Hunt (or by admins manually), the system **auto-generates** a username and stores it in a new column `directory_profiles.unclaimed_handle TEXT UNIQUE NULL`.
- Auto-generated handles use the reserved prefix `@community-<6-char-hex>` (e.g., `@community-7f3a2b`). Clerk must be configured to **block** this prefix so a real user can never claim a colliding handle.
- When a real user joins and claims an unclaimed profile, `directory_profiles.claimed_by_user_id` is set to their user_id and the public URL resolver switches from `unclaimed_handle` → `users.username`. The `unclaimed_handle` value is kept for historical/audit purposes but the URL no longer uses it.
- This avoids any interference with Clerk's auth logic. Clerk owns real usernames; our DB owns the temporary `@community-...` placeholders.

### 2.3 Rare skill detection
**Live read from Workforce plugin.** Rare = skill where <50% of Workforce profiles claiming that skill have `recruited_state = TRUE`. Computed at round-create time and **snapshotted** into `skills_hunt_rare_skills_lookup` so leaderboards do not shift mid-round if Workforce data moves. Admins can re-snapshot manually if needed.

### 2.4 Submission modal location
**Primary entry point is the Directory public page**, via a "Submit a community profile" CTA on the pinned reward card. The modal lives in `components/skills-hunt/` and is imported by `components/directory/directory-shell.tsx`. This maximizes the funnel: every visitor browsing the Directory sees the way to contribute. A secondary entry point lives on the Skills Hunt plugin page.

### 2.5 Backfill for existing 60 Directory profiles
On the first deploy of `unclaimed_handle`, run a one-shot migration that assigns `@community-<6-char-hex>` handles to every existing Directory profile that has `claimed_by_user_id IS NULL`. This makes the @handle URL story consistent on day one.

## 3. Audit findings (2026-05-11 snapshot)

The plugin was already partially implemented when this rewrite started. The audit ran across schema, lib, API, UI, mobile, contracts, and inventory. Full findings preserved in commit history (`git log` on `claude/audit-skills-hunt-plugin-6yv3e`).

### 3.1 What works
- Core schema tables exist: `skills_hunt_rounds`, `_submissions`, `_leaderboard`, `_achievements`, `_notifications`, `_feature_reward_card`, `_rare_skills_lookup`, `_directory_profiles`, `_audit_log`, `_service_credits_transactions`.
- All major HTTP routes are present under `/api/skills-hunt/*` for both user and admin surfaces.
- Submission creation, duplicate detection (signature_hash), rolling rate limit (10/7d), and Quora URL format validation are working.
- Admin review action (`POST /api/skills-hunt/admin/submissions/{id}/review`) accepts/rejects/edits/flags and persists outcomes.
- Unclaimed Directory profile auto-generation on accept works through `maybeAutoGenerateDirectoryProfile()`.
- Audit log is solid (append-only, allow/deny tracked).
- Contracts (command/access-policy/audit YAMLs) exist and align with implemented commands.
- Plugin registry entry exists in `lib/plugins/repository.ts`.

### 3.2 What's missing or wrong (the rewrite list)

**Schema gaps:**
- `skills_hunt_submissions` is missing: `url_validation_result`, `credit_granted`, `proposed_skills`, `edit_history`, `edited_at`, `deleted_at`, `participation_points`.
- `skills_hunt_leaderboard` is missing: `first_match_count` (needed for tie-break), `pending_points`, `last_submission_at`.
- `skills_hunt_achievements` is missing: `round_id` FK, `archived_at`.
- `directory_profiles` is missing: `source` enum (admin/self/community-generated), `invited_by_username`, `unclaimed_handle UNIQUE`, `deleted_at`.
- No `skills_hunt_submission_reports` table for moderation/report flow.

**Validation:**
- URL HEAD-check (liveness) is NOT implemented. Spec requires HEAD request with 5s timeout; persist `url_validation_result` ∈ {valid, invalid, dead}.
- Submitter username can be NULL (`requireUsername: false` on submit endpoint). Must flip to `true` once Clerk onboarding gate is in.
- Reputation system is NOT implemented (new-user lower limits; >20% rejection → admin pre-approval; ≥80% acceptance → raised limits).

**Scoring engine:**
- Implemented weights diverge from spec: match is `min(skills.length,5)*3` not `+10`; stack is linear not gated; first match is `+4` not `+5`; quality is `bio≥200chars` not `no admin edits`; participation point on reject is missing.
- Workforce live read for rare-skill is NOT wired — currently a manually-curated `skills_hunt_rare_skills_lookup` table.

**Badges:**
- Only 3 generic count-based badges exist (`accepted-first`, `accepted-five`, `accepted-ten`).
- Spec requires 5 named badges: First Finder, Diversity Champion, Rare Talent Scout, Quality Contributor, Leaderboard Champion. None implemented.

**Leaderboard:**
- No tie-break logic (spec: `accepted_points DESC, first_match_count DESC, submission_date ASC`).
- No team mode aggregation.
- No all-time view.
- No live updates (websocket/polling).
- No top-100 + your-rank cap on response.

**Notifications:**
- GetStream.io is integrated in the repo (`lib/lighthouse/stream.ts`, `lib/shared/stream-feeds.ts`) but Skills Hunt does NOT fan out through it — only in-DB notifications.
- Only `submission-accepted` and `submission-rejected` are emitted. Missing: leaderboard-update (top 10), round-ending-soon (24h), achievement-unlocked.
- No notification-center UI on web or mobile.

**Directory integration:**
- "Community generated profile" label is stored only in `skills_hunt_directory_profiles.metadata = {"generatedBy":"skills-hunt"}`. Not visible on the public profile page.
- `invited_by_username` lives only in the junction table, not on `directory_profiles` itself. Admin-created unclaimed profiles get no attribution column.
- No `@handle` URL routing — only `/apps/directory/{id}`.

**Security / compliance:**
- No soft-delete columns; uses CASCADE hard deletes.
- No GDPR delete endpoint (`DELETE /api/account/skills-hunt-profile`).
- No moderation/report flow ("report this profile, permission not given").
- No bulk admin actions, no CSV export, no dispute escalation.

**UI surfaces:**
- **Submission modal does not exist** as a React component.
- Reward card is admin-configured but NOT rendered on Directory public page.
- Notification center UI does not exist.
- Admin panel is read-only summary; no inline accept/reject/edit, no filters, no bulk actions.
- Badges are not rendered on user Directory profile.

**Mobile:**
- `packages/mobile/src/features/skills-hunt/SkillsHunt.tsx` is a 143-line hardcoded mock with no API calls.

**Validation limit drift (need alignment before modal is built):**
| Field | Spec | Currently | Decision |
| ---- | ---- | ---- | ---- |
| Display name | 2–100 chars, alphanumeric+spaces | 1–120, no charset | Spec wins: 2–100, alphanumeric + spaces |
| Bio | max 280 | 1–1200 | Spec wins: max 280 chars (one-sentence intent) |
| Skills | 1–140 chars text blob | 1–25 items | Owner override: multi-select from taxonomy + optional free-text parse to chips (max 10 chips, each ≤40 chars) |

## 4. Companion plugin: Directory username + URL changes

Skills Hunt cannot ship without these Directory plugin changes. They are **owned by the Directory plugin** but tracked here because Skills Hunt blocks on them.

### 4.1 New columns on `directory_profiles`
1. `source TEXT NOT NULL DEFAULT 'admin' CHECK (source IN ('admin', 'self', 'community-generated'))` — marks how the profile entered the system. Drives the "community generated profile" badge in UI.
2. `invited_by_username TEXT NULL` — denormalized from `skills_hunt_directory_profiles.invited_by_username` so the Directory profile page can render attribution without joining.
3. `unclaimed_handle TEXT UNIQUE NULL` — auto-generated vanity handle for unclaimed profiles. Format: `community-<6char-hex>` (no leading `@` in storage; the `@` is presentation only).
4. `deleted_at TIMESTAMPTZ NULL` — soft-delete flag for GDPR + moderation removals.

### 4.2 URL routing
- Canonical URL for a claimed profile: `/apps/directory/@{users.username}`.
- Canonical URL for an unclaimed profile: `/apps/directory/@{directory_profiles.unclaimed_handle}`.
- Legacy `/apps/directory/{uuid}` issues a 301 redirect to the canonical `@handle` URL.
- Handle resolver lookup order: try `users.username` → if found, resolve to that user's directory profile. Else try `directory_profiles.unclaimed_handle`.

### 4.3 Clerk reserved prefix
Clerk dashboard must reject usernames starting with `community-`. We also enforce this at our API layer as defense in depth (in case Clerk config drifts).

### 4.4 Backfill migration
One-shot: for every `directory_profiles` row where `claimed_by_user_id IS NULL AND unclaimed_handle IS NULL`, assign `unclaimed_handle = 'community-' || encode(gen_random_bytes(3), 'hex')`. Retry on UNIQUE collision. Run idempotently.

## 5. Architectural rules that bind future agents

1. **No automated scraping.** Quora liveness check uses HEAD only (no body fetched). No cron job pulling profile data. Owner has stated this is non-negotiable (ToS + ethics).
2. **Directory boundary.** Skills Hunt may only create unclaimed Directory profiles through `lib/directory/repository.ts` exports. Skills Hunt MUST NOT directly mutate `directory_profiles` rows. Ownership claim lifecycle remains Directory-authoritative.
3. **Audit everything.** Every admin mutation writes to `skills_hunt_audit_log`. Use the existing `insertSkillsHuntAudit` helper.
4. **Server-side authorization is source of truth.** Frontend hiding is presentation-only.
5. **Schema migrations** follow CLAUDE.md: use `CREATE TABLE IF NOT EXISTS` listing all current columns AND `ALTER TABLE IF EXISTS ADD COLUMN IF NOT EXISTS` for every new column. This serves both fresh and legacy DBs.
6. **Type safety.** No unjustified `any`. If unavoidable, use `eslint-disable-next-line @typescript-eslint/no-explicit-any` with comment + TODO.
7. **Inventory sync** is mandatory on every code change per CLAUDE.md "Plugin Feature Inventory Sync Policy."
8. **Commit messages** must end with the active Claude session URL on its own line.
9. **Branch discipline.** All Skills Hunt rewrite work goes to `claude/audit-skills-hunt-plugin-6yv3e` until the design pass is complete.
10. **Build verification.** Run `pnpm build` after every coherent slice. Do not push a broken tree.

## 7. Open questions still pending owner decision

These should be resolved before the corresponding Wave 2 work begins.

1. **Leaderboard live updates**: WebSocket or polling? GetStream feeds support real-time push but adds infra cost. Lightweight 30s polling is simpler. **Recommendation:** start with polling; upgrade to WebSocket if engagement metrics justify.
2. **Moderation report flow placement**: report button on every Directory profile, or only on community-generated ones? **Recommendation:** all profiles, but escalation path differs.
3. **Dispute escalation**: second-admin sign-off, or just a flagged queue the owner reviews personally? **Recommendation:** flagged queue for now; introduce second-admin only if volume justifies.
4. **CSV export**: ad-hoc download endpoint, or scheduled exports to a storage bucket? **Recommendation:** ad-hoc on-demand endpoint.
